# Restaurant Booking System вЂ” Implementation Plan

## Scope Boundaries

**In scope:** Restaurant list, booking form, booking confirmation, existing bookings, availability query, conflict prevention, full validation, typed OpenAPI client, polished UI.

**Out of scope:** Auth/users, payments, email notifications, durable persistence (in-memory only), admin CRUD for restaurants/tables (use seed data), rate limiting, cancellation, edit bookings.

## Architecture Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Backend framework | .NET 10 Minimal APIs | Required. Use `TypedResults`, built-in validation. |
| Persistence | In-memory `ConcurrentDictionary` | Fastest path to working behavior; no EF Core distraction. |
| Validation | Built-in DataAnnotations (`AddValidation()`) | Zero-config for simple DTOs. |
| Result pattern | Custom `Result<T>` / `Error` types | Avoids exceptions for business failures (conflict, not found). |
| FCIS | Pure domain functions in `Domain/`; thin endpoint handlers in `Features/` | Testable logic without I/O mocks. |
| Project structure | Vertical slices (feature folders) | Each domain operation is a self-contained folder. |
| OpenAPI | `MapOpenApi()` + Swagger UI | Required for Orval client generation. |
| Frontend framework | Vite + React 19 + TypeScript | Fast SPA setup. |
| UI kit | shadcn/ui + Tailwind CSS v4 | Required. Component-driven, accessible. |
| API client | Orval from OpenAPI spec | Fully typed, no hand-written fetch. |
| State management | TanStack Query | Server state caching, mutations, auto-refetch. |

## Implementation Order (Vertical TDD Slices)

### Slice 1: Backend Skeleton + OpenAPI
1. Create .NET 10 project with `<TreatWarningsAsErrors>true</TreatWarningsAsErrors>`, `LangVersion=14`, `Nullable=enable`
2. Configure `AddValidation()`, `MapOpenApi()`, Swagger UI, CORS
3. Add seed data (3 restaurants, 3-5 tables each)
4. In-memory store (`ConcurrentDictionary`)
5. Verify OpenAPI document at `/openapi/v1.json`

### Slice 2: Domain вЂ” Pure Functions (Tests First)
1. `ValidationRules.cs` вЂ” `ValidatePartySize(int)`, `ValidateDateTime(DateTime)`, `ValidateRestaurantExists(Guid)`
2. `BookingConflict.cs` вЂ” `HasOverlap(Booking, IEnumerable<Booking>)`
3. `Availability.cs` вЂ” `GetAvailableSlots(restaurant, date, partySize, existingBookings)`
4. Write boundary tests for each (invalid party size -1, 0, 1000; same-time conflict; adjacent time slot not conflicting)

### Slice 3: Create Booking Endpoint + Tests
1. `CreateBooking.cs` вЂ” endpoint with DTO, handler calling domain functions
2. Integration tests via `WebApplicationFactory`: happy path, conflict, invalid party size, unknown restaurant, invalid date
3. Return `TypedResults` with proper status codes (201, 400, 404, 409)

### Slice 4: List Endpoints
1. `ListRestaurants.cs` вЂ” returns all restaurants with table info
2. `ListBookings.cs` вЂ” returns bookings, optionally filtered by restaurant/date
3. `GetAvailability.cs` вЂ” returns time slots for a restaurant+date+partySize

### Slice 5: Frontend Skeleton + API Client
1. Create Vite project with React, TypeScript, Tailwind CSS v4
2. Initialize shadcn/ui with components.json
3. Configure Orval (`orval.config.ts` pointing at backend OpenAPI spec)
4. Run `generate-api` script вЂ” produce `src/api/booking-api.ts`
5. Add button, card, input, label, select, table, badge shadcn components

### Slice 6: Restaurant List + Booking Form
1. `use-bookings.ts` вЂ” TanStack Query hooks (useRestaurantList, useCreateBooking, useBookingList, useAvailability)
2. `RestaurantList.tsx` вЂ” card grid with name, description, table count
3. `BookingForm.tsx` вЂ” restaurant select, date picker, time dropdown, party size input, guest name, guest email
4. Wire form to `useCreateBooking` mutation

### Slice 7: Booking Confirmation + Booking List
1. `BookingConfirmation.tsx` вЂ” shows after successful booking, links to bookings list
2. `BookingList.tsx` вЂ” TanStack Table or simple table with restaurant, date, time, party size, guest name
3. `AvailabilityView.tsx` вЂ” visual time slot grid with available/booked states

### Slice 8: Polish
1. Responsive layout (mobile-first cards, desktop table)
2. Accessible labels, focus management on form submission
3. Error states (toast or inline for conflicts, validation errors)
4. Loading skeletons / spinners
5. Dead code elimination pass
6. Run all lint/format/typecheck/build scripts

## Validation Rules (Domain)

| Rule | Error |
|------|-------|
| Party size 1-20 | `InvalidPartySize` |
| Booking date/time not in the past | `InvalidDateTime` |
| Restaurant must exist | `RestaurantNotFound` |
| Table exists for restaurant | `TableNotFound` |
| Time slot available (no overlap on same table) | `BookingConflict` |
| Booking duration: 90 minutes fixed | (implicit in slot calculation) |

## Seed Data

3 restaurants with operating hours 11:00вЂ“22:00, tables with capacities [2, 2, 4, 4, 6, 8]:
- "Trattoria Milano" вЂ” Italian, 6 tables
- "Sakura Garden" вЂ” Japanese, 5 tables
- "Le Bistro Parisien" вЂ” French, 4 tables

## Time Slot Model

- 30-minute increments from 11:00 to 21:00 (last seating)
- 90-minute fixed booking duration
- A slot is available if: table capacity >= party size AND no overlapping booking exists

## Orval Configuration

```ts
// orval.config.ts
export default {
  booking: {
    input: '../backend/src/RestaurantBooking/openapi.json', // or fetch from running server
    output: {
      target: './src/api/booking-api.ts',
      client: 'react-query',  // generates TanStack Query hooks
      mode: 'tags-split',     // one file per tag (optional)
      override: {
        mutator: './src/lib/api-client.ts',
      },
    },
  },
};
```

Generate script: `curl http://localhost:5000/openapi/v1.json -o openapi.json && npx orval`

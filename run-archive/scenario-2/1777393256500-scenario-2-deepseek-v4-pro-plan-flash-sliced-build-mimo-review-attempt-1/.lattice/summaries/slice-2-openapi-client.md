## How The Custom Fetch Mutator Works

1. The SPA calls `GET /api/auth/csrf` to get the CSRF token (returned both in response body and `X-CSRF-TOKEN` header)
2. The SPA stores the token via `setCsrfToken(token)` from `src/lib/api.ts`
3. Every subsequent Orval-generated request automatically includes:
   - `credentials: "include"` for the auth cookie
   - `X-CSRF-TOKEN` header (if token is set)
4. On logout, the SPA clears the token with `setCsrfToken(null)`

## Known Gaps

- The checked-in OpenAPI spec is hand-crafted to be cleaner and more Orval-compatible than the live auto-generated spec (e.g., `"type": "integer"` instead of `"type": ["integer", "string"]`). This is intentional — the checked-in spec is the source of truth for client generation.
- OperationIds in the checked-in spec use camelCase (`listRestaurants`) while the backend auto-generates PascalCase (`ListRestaurants`). This doesn't affect functionality since the checked-in spec drives client generation.
- App.tsx still uses the raw fetch functions — Slice 3 will migrate to the generated hooks.

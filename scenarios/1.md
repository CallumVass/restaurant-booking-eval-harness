# Restaurant Booking System Eval

Build a restaurant booking system.

## Backend

- Use .NET 10 Web API.
- Model restaurants, tables, and bookings.
- Create a booking endpoint with conflict prevention.
- List available time slots for a restaurant, date, and party size.
- Validate invalid party size, invalid dates/times, unknown restaurants, and overlapping reservations.
- Enable OpenAPI/Swagger.

## Frontend

- Use a React SPA.
- Show a restaurant list.
- Provide a booking form.
- Show booking confirmation.
- Show existing bookings.
- Integrate with the API through clear client-side service functions.

## Engineering Preferences

- Work in vertical TDD slices where practical.
- Prefer pure domain functions and thin imperative shells.
- Use explicit Result-style errors for expected business failures.
- Keep Clean Architecture boundaries lightweight and appropriate for a 30-60 minute eval.
- Include boundary tests for booking conflicts, invalid party size, invalid times, and overlapping reservations.
- Configure deterministic frontend scripts: `build`, `typecheck`, `lint`, `format:check`, and `deadcode`.
- Keep backend and frontend formatted, linted, warning-free, and free of dead code or unused exports.
- Include README run instructions.

## Constraints

- Keep scope realistic for a 30-60 minute implementation.
- Prefer in-memory persistence unless durable persistence is quick and does not distract from core behavior.
- Ensure backend build passes with warnings treated as errors.
- Ensure backend tests pass without rebuilding after the build step and include boundary tests.
- Ensure backend formatting verification passes.
- Ensure frontend dependencies install and `build`, `typecheck`, `lint`, `format:check`, and `deadcode` scripts pass.

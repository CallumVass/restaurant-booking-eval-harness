# Enterprise Owner/Admin Compliance Hardening

Extend the existing authenticated restaurant booking app as a brownfield enterprise compliance change. This scenario starts from the Scenario 3 baseline plus a large compliance pack. Do not rebuild the app from scratch.

## Required Source Of Truth

Before editing, inspect and reconcile the full compliance pack in the baseline:

- `docs/compliance/authz-matrix.md`
- `docs/compliance/audit-redaction-policy.md`
- `docs/compliance/legacy-contracts.md`
- `backend/RestaurantBooking.Api/ComplianceData/scenario4-compliance-fixtures.json`

The files are intentionally large and contain repeated examples plus later reminders. Treat their canonical account, owner-mapping, audit-action, redaction, and legacy-route requirements as authoritative. Preserve meaningful compliance markers from the pack in backend tests or source where they prove behavior.

## Goal

Implement role-based restaurant owner/admin operations and audit hardening while preserving the existing diner auth/booking behavior.

Roles:

- `diner`: browses restaurants, checks availability, creates bookings, and views only their own booking history.
- `restaurant-owner`: manages only owned restaurants and views bookings only for those restaurants.
- `admin`: views owner mappings, admin-scoped bookings, and safe audit events.

Seed and document at least these local accounts:

- `diner.demo@example.test`
- `owner.ember@example.test` owns `ember-table` (`RBAC-S4-OWNER-EMBER`)
- `owner.luna@example.test` owns `luna-verde` (`RBAC-S4-OWNER-LUNA`)
- `owner.saffron@example.test` owns `saffron-court` (`RBAC-S4-OWNER-SAFFRON`)
- `admin.ops@example.test`

Passwords may be local-development friendly but must be documented and must not appear in audit/admin responses.

## Authorization Requirements

- Unauthenticated protected requests return `401`.
- Authenticated users without the required role or ownership return `403`.
- Owners can access only their mapped restaurants and bookings.
- Wrong-owner access must be `403`, not empty `200`.
- Diners cannot access owner/admin endpoints.
- Admin global visibility must be exposed only through clearly admin-scoped routes, not by broadening diner routes.
- `/api/bookings` remains diner-scoped.
- `/api/restaurants/{restaurantId}/bookings` remains diner-scoped unless owner/admin-specific routes are clearly separated.

## Owner/Admin Features

Owners need:

- owned restaurant list,
- owned restaurant detail,
- owner-scoped restaurant profile update for description/neighborhood/cuisine,
- owner-scoped restaurant bookings.

Admins need:

- owner mapping view,
- safe audit event view,
- admin-scoped all-bookings view if needed.

Do not add table-capacity mutation unless you add validation/tests that prove existing bookings are not silently invalidated.

## Audit Requirements

Every successful required state-changing operation writes a safe audit event using exact action names from the compliance pack, including at least:

- `user.registered.v2`
- `user.login.succeeded.v2`
- `user.logout.succeeded.v2`
- `booking.created.v2`
- `owner.restaurant.profile.updated.v2`

Audit events include event id, UTC timestamp, actor id/email/role, action, target type/id, outcome, and bounded safe metadata.

Audit storage and admin responses must never include denylisted sensitive fields from the compliance pack, including password/plaintext/hash data, CSRF/XSRF tokens, cookies/set-cookie headers, authorization headers, security stamps, reset/refresh tokens, raw request headers, full claims, session ids, API keys, or connection strings.

## Backend Requirements

- Keep .NET 10 minimal APIs.
- Keep booking/availability rules pure where practical.
- Keep auth, authorization, audit, and HTTP wiring explicit.
- Add clear owner/admin routes with OpenAPI names/metadata and `401`/`403` responses.
- Preserve CSRF protection for all state-changing cookie-auth requests.
- Update live OpenAPI and checked-in frontend OpenAPI source.
- Add HTTP tests across unauthenticated, diner, owner Ember, owner Luna, owner Saffron, and admin.
- Add tests proving exact owner mapping canaries, exact audit action names, and redaction denylist behavior.

## Frontend Requirements

- Preserve React, Tailwind/shadcn style, TanStack Query, and Orval-generated client workflow.
- Use generated operations/hooks for new owner/admin/audit endpoints where supported.
- Add role-aware diner, owner, and admin panels.
- Hide unavailable role UI, but rely on backend authorization for security.
- Show clear `401`/`403` states.
- Render audit events as safe summaries/tables, not raw JSON dumps.
- Add focused frontend tests for role navigation, owner/admin panels, permission errors, safe audit display, and logout clearing role UI.

## Constraints

- No external hosted auth, OAuth apps, external databases, or secrets.
- Keep data in-memory/local unless the existing app already uses another approach.
- Do not weaken HTTP-only cookie auth or CSRF protection.
- Do not store auth tokens in `localStorage` or `sessionStorage`.
- Do not expose secrets or raw security material through audit/admin APIs.
- Prefer focused edits over replacing project structure.

# Add Restaurant Owner/Admin Authorization And Audit Logging

Extend the existing authenticated restaurant booking app. Treat this as a normal brownfield change request against the current Scenario 2 codebase: inspect the backend, frontend, tests, OpenAPI/client generation, scripts, and README before changing anything.

Do not rebuild the app from scratch. Preserve existing diner booking behavior unless this request explicitly changes it.

## Goal

Add role-based restaurant operations and audit hardening on top of the existing local cookie-auth booking system.

The app now needs three local account roles:

- `diner`: can browse restaurants, create their own bookings, and view only their own booking history.
- `restaurant-owner`: can manage only the restaurants they own and view bookings for only those restaurants.
- `admin`: can view all restaurants, owners, and audit events, but must not see password hashes, CSRF tokens, auth cookies, or raw secrets.

Keep auth local-development friendly. Seeded users are acceptable and should be documented.

## Existing Behavior To Preserve

- Public users can still list restaurants and check availability.
- Authenticated diners can still create bookings with CSRF protection.
- Diners can still view their own booking history and cannot view another diner's history.
- Existing booking validation, table capacity, invalid date/time, unknown restaurant, and overlap/conflict behavior must continue to work.
- Existing generated OpenAPI client workflow, TanStack Query usage, Tailwind/shadcn UI style, deterministic scripts, and README structure should remain recognizable.

## New Authorization Requirements

- Add role/ownership data to local users.
- Seed at least:
  - one diner demo account,
  - one owner account for one restaurant,
  - one owner account for a different restaurant,
  - one admin account.
- A restaurant owner can view bookings for restaurants they own.
- A restaurant owner cannot view or modify another owner's restaurants or bookings.
- A diner cannot access owner-only or admin-only endpoints.
- An admin can view all restaurant-owner mappings and audit events.
- Admin access must not bypass diner booking ownership rules in the diner-facing history endpoints; if admin needs all-bookings visibility, expose a clearly admin-scoped endpoint.
- Protected endpoints must return appropriate `401` for unauthenticated requests and `403` for authenticated users without permission.

## Restaurant Management Requirements

Add owner-scoped restaurant management without replacing the current in-memory architecture.

Owners should be able to manage only their owned restaurants:

- view owned restaurant details,
- update restaurant description/neighborhood/cuisine or similar editable profile fields,
- view bookings for their owned restaurant.

Do not allow owners to change table capacities in a way that silently invalidates existing bookings unless you add validation/tests for that rule. It is acceptable to keep table management out of scope and document it as intentionally unsupported.

## Audit Requirements

Every successful state-changing operation must write an audit event, including at least:

- user registration,
- login,
- logout,
- booking creation,
- owner restaurant profile update.

Audit events must include:

- event id,
- timestamp,
- actor user id/email/role where available,
- action name,
- target type/id where applicable,
- outcome (`success` is enough for required logged events; failed attempts may be logged if implemented),
- safe metadata useful for debugging.

Audit events must not include:

- plaintext passwords,
- password hashes,
- CSRF tokens,
- auth cookie values,
- raw request headers,
- full auth claims dumps.

Admins can view audit events through a protected admin endpoint. Diners and owners cannot view audit events.

## Backend Requirements

- Keep using .NET 10 minimal APIs.
- Keep the functional core for booking/availability rules pure where practical.
- Keep imperative auth, authorization, audit, and HTTP wiring thin and explicit.
- Use explicit result/error types for expected ownership/authorization/business failures where practical.
- Add owner/admin endpoints with clear route names and OpenAPI metadata.
- Update live OpenAPI and checked-in frontend OpenAPI source to include role/owner/admin/audit endpoints and `401`/`403` responses.
- Preserve CSRF protection for all state-changing cookie-auth requests.
- Add tests for authorization boundaries across diner, owner A, owner B, admin, and unauthenticated clients.
- Add tests proving audit events are written for required successful state-changing operations and do not leak password/token/cookie material.

## Frontend Requirements

- Keep the existing React SPA, Tailwind CSS, shadcn/ui approach, TanStack Query, and generated OpenAPI client workflow.
- Continue using generated TanStack Query hooks where Orval supports them.
- Add role-aware navigation or panels:
  - diner booking flow/history,
  - owner restaurant management/history panel,
  - admin audit panel.
- Hide owner/admin UI from users without those roles, but do not rely on frontend hiding for security.
- Show clear permission/error states for `401`/`403` responses.
- Keep UI polished and responsive; avoid dumping raw JSON for the admin audit view.
- Add focused frontend tests for role-specific UI and permission errors where practical.

## Tests And Quality Gates

Work in small slices: inspect, plan, implement, verify.

Required backend tests should cover:

- seeded role accounts can log in,
- unauthenticated owner/admin endpoints return `401`,
- diner access to owner/admin endpoints returns `403`,
- owner A can access owner A restaurant bookings/profile,
- owner A cannot access owner B restaurant bookings/profile,
- admin can view audit events and owner mappings,
- audit events are created for registration/login/logout/booking creation/owner profile update,
- audit responses do not contain passwords, password hashes, CSRF tokens, or cookie values,
- existing diner booking ownership and conflict/validation behavior still works.

Required frontend tests should cover where practical:

- diner booking flow still works,
- owner panel appears only for owner accounts,
- owner permission error is displayed for unauthorized restaurant access,
- admin audit panel displays safe audit event summaries,
- logout clears role-specific UI.

- frontend tests if a test script exists or you add one.

## Constraints

- No external hosted auth, OAuth apps, external databases, or secrets.
- Keep data in-memory/local unless the existing app already uses another approach.
- Do not weaken HTTP-only cookie auth or CSRF protection.
- Do not store auth tokens in `localStorage` or `sessionStorage`.
- Do not expose password hashes, tokens, cookies, or raw headers through audit/admin APIs.
- Prefer focused edits over replacing project structure.

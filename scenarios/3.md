# Add Authenticated Booking Accounts

Extend the existing restaurant booking app so bookings belong to authenticated users. Treat this as a normal change request against the current codebase: inspect the existing backend, frontend, tests, OpenAPI/client generation, scripts, and README before changing anything.

Do not rebuild the app from scratch. Preserve existing behavior unless this request explicitly requires a change.

## Existing Issues To Fix

- Make booking conflict prevention atomic across validation and insert/update. Concurrent booking requests must not be able to double-book the same table.
- Add regression tests that prove concurrent or near-concurrent overlapping booking attempts cannot both succeed.
- Add focused frontend tests for the core booking/auth flows where practical.
- Improve the booking form integration with the existing UI system, especially form fields, selection controls, validation display, and accessible error states.

Keep fixes focused. Preserve the existing architecture, generated API client workflow, TanStack Query usage, Tailwind/shadcn setup, and deterministic scripts.

## Authentication

- Add local email/password authentication using ASP.NET Core Identity or an equivalent first-party .NET auth approach.
- Do not use external hosted auth providers, OAuth apps, or services requiring secrets.
- A user must be authenticated to create a booking.
- A user can view their own booking history at restaurants.
- Users must not be able to view another user's booking history.
- Keep auth suitable for local development and automated verification; seeded/demo users are acceptable if documented.

## Cookie Auth Security

- Use HTTP-only cookies for authentication.
- Because auth uses cookies, implement local-development appropriate CSRF/anti-forgery protection for state-changing requests.
- The SPA must fetch and send the CSRF token for login, logout, booking creation, and other booking mutations.
- Configure CORS and credentialed requests deliberately for the local frontend/backend origins.
- Do not store auth tokens in localStorage or sessionStorage.

## Backend Requirements

- Keep using .NET 10 Web API.
- Keep existing restaurants, tables, availability, and booking behavior working.
- Protect booking creation so unauthenticated users cannot create bookings.
- Associate each booking with the authenticated user that created it.
- Add authenticated booking-history endpoints scoped to the current user.
- Preserve useful validation errors for invalid party size, invalid dates/times, unknown restaurants, and overlapping reservations.
- Keep OpenAPI/Swagger usable for frontend client generation, including auth-related endpoints and protected booking behavior.

## Frontend Requirements

- Keep using the existing React SPA, Tailwind CSS, shadcn/ui approach, TanStack Query, and generated OpenAPI client workflow.
- Add login, registration, logout, and authenticated-user state to the existing UI.
- Require login before creating a booking and show a clear path to authenticate.
- Show the authenticated user's booking history at restaurants.
- Use the generated API client and TanStack Query/mutations for auth and booking flows where practical.
- Avoid hand-written, stringly typed fetch wrappers when an OpenAPI-generated typed client is available.
- Keep the UI polished, responsive, accessible, and consistent with the existing visual language.

## Tests And Quality Gates

- Work in small slices: inspect, test, change, verify.
- Prefer minimal focused edits over replacing existing structure.
- Prefer pure domain functions and thin imperative shells for booking/conflict logic.
- Use explicit Result-style errors for expected business failures where the existing app does so.
- Keep Clean Architecture boundaries lightweight and consistent with the existing app.
- Include backend tests for auth boundaries, user-scoped booking history, invalid auth attempts, atomic conflict prevention, invalid party size, invalid times, and overlapping reservations.
- Include frontend tests for the main auth and booking-history flows where practical.
- Keep backend and frontend formatted, linted, warning-free, and free of dead code or unused exports.
- Update README run instructions for auth, seeded/demo users, CSRF behavior, and any new scripts.

## Constraints

- Keep scope realistic for a focused implementation pass.
- Do not introduce external services, hosted databases, OAuth providers, or secrets.
- Prefer in-memory or lightweight local persistence unless the existing app already uses another approach.
- Ensure backend build passes with warnings treated as errors.
- Ensure backend tests pass without rebuilding after the build step and include the required auth/conflict boundary tests.
- Ensure backend formatting verification passes.
- Ensure frontend dependencies install and `build`, `typecheck`, `lint`, `format:check`, and `deadcode` scripts pass.
- Ensure typed-client generation remains reproducible from source, either as part of frontend setup/build or via a documented script used by the build.
- Follow the saved implementation plan closely; if changing direction, document why in the README or final notes.
